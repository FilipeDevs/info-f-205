# Exercice 14 — Représentation hexadécimale et IEEE 754

---

## 🧱 Brique 1 — Comment un nombre est stocké en mémoire (concrètement)

Imagine que la mémoire de l'ordinateur est une longue rangée de cases,
chaque case contenant soit un 0 soit un 1 (un "bit").

Pour un nombre flottant en **double précision** (le standard des ordinateurs
modernes), on utilise **64 cases** (64 bits), découpées en 3 zones :

```
┌──────┬──────────────────────┬─────────────────────────────────────────────────┐
│Signe │      Exposant        │                  Mantisse                       │
│1 bit │      11 bits         │                  52 bits                        │
└──────┴──────────────────────┴─────────────────────────────────────────────────┘
```

> 💡 **Analogie concrète :** C'est comme écrire un nombre en notation
> scientifique sur une fiche cartonnée :
> - Une case pour le signe (+/-)
> - Une case pour la puissance (l'exposant)
> - Une case pour les chiffres (la mantisse)
>
> **Exemple :** $-3.14 \times 10^5$ → signe="-", exposant="5", mantisse="314"

## 🧱 Brique 2 — C'est quoi l'hexadécimal ?

L'hexadécimal (base 16) est juste une façon **compacte** d'écrire des bits.
Au lieu d'écrire 64 zéros et uns (illisible), on les regroupe par 4 :

```
4 bits = 1 chiffre hexadécimal
```

| Bits | Hex | | Bits | Hex |
|---|---|---|---|---|
| 0000 | 0 | | 1000 | 8 |
| 0001 | 1 | | 1001 | 9 |
| 0010 | 2 | | 1010 | A |
| 0011 | 3 | | 1011 | B |
| 0100 | 4 | | 1100 | C |
| 0101 | 5 | | 1101 | D |
| 0110 | 6 | | 1110 | E |
| 0111 | 7 | | 1111 | F |

Donc 64 bits = 16 chiffres hexadécimaux. C'est pour ça que Matlab affiche
16 caractères dans `format hex`.

> 💡 **Exemple :**
> `400921fb54442d18` = 16 chiffres hex = 64 bits en mémoire

## 🧱 Brique 3 — Pourquoi réserver les exposants extrêmes ?

Avec 11 bits pour l'exposant, on peut former $2^{11} = 2048$ combinaisons,
de `00000000000` (tout à 0) à `11111111111` (tout à 1).

**Pourquoi ne pas les utiliser toutes pour des nombres normaux ?**

On a besoin de représenter des **valeurs spéciales** :
- **0** (le zéro) → pas représentable avec la règle normale (mantisse commence
  toujours par 1)
- **$+\infty$ et $-\infty$** → quand un calcul déborde (overflow)
- **NaN** (Not a Number) → résultat d'opérations impossibles comme $0/0$

La convention IEEE 754 est :
- Exposant `00000000000` (tout à 0) → **réservé pour le zéro** et les
  nombres dénormalisés (très petits)
- Exposant `11111111111` (tout à 1) → **réservé pour $\pm\infty$ et NaN**

> 💡 **Analogie :** C'est comme réserver les numéros de téléphone
> commençant par 15 (SAMU), 17 (Police), 18 (Pompiers). Ces numéros
> existent mais ont une signification spéciale — on ne les attribue pas
> à des abonnés normaux.

## 🧱 Brique 4 — Comment le signe fonctionne dans la représentation hex

Le **tout premier bit** de la représentation 64 bits est le bit de signe :
- `0` → nombre positif
- `1` → nombre négatif

En hexadécimal, le premier chiffre hex correspond aux **4 premiers bits**,
donc le premier bit de signe est le **premier bit du premier chiffre hex**.

```
π = 400921fb54442d18
    ↑
    4 = 0100 en binaire
    ↑
    premier bit = 0 → positif ✓

-π = c00921fb54442d18
     ↑
     c = 1100 en binaire
     ↑
     premier bit = 1 → négatif ✓
```

**Seul le premier chiffre hex change** (4 → c), tout le reste est identique !

## 🧱 Brique 5 — Pourquoi π et π+1 ont des représentations très différentes

La clé est dans le mot **"flottante"** : la virgule **flotte** selon l'exposant.

Regardons où se trouvent $\pi$ et $\pi + 1$ par rapport aux puissances de 2 :

```
    2¹=2        π≈3.14       2²=4     π+1≈4.14      2³=8
     |             |          |          |            |
─────┼─────────────┼──────────┼──────────┼────────────┼─────
```

- $\pi \approx 3.14$ est entre $2^1$ et $2^2$ → **exposant = 1**
  $$\pi = 1.5707... \times 2^1$$

- $\pi + 1 \approx 4.14$ est entre $2^2$ et $2^3$ → **exposant = 2**
  $$\pi + 1 = 1.0353... \times 2^2$$

Les deux nombres ont des exposants différents → la virgule est à un endroit
différent → **les 52 bits de la mantisse représentent des plages complètement
différentes** → la représentation hex change complètement.

> 💡 **Analogie :** C'est comme écrire 3.14 et 4.14 en notation scientifique.
> Même si les nombres diffèrent de 1, les mantisses sont totalement différentes :
> $1.57 \times 2^1$ vs $1.035 \times 2^2$. La virgule a "flotté" d'une position.

---

## 🔧 Recette à l'examen

**Pour changer le signe dans la repr. hex :**
1. Convertir le premier chiffre hex en binaire (4 bits)
2. Inverser le premier bit (0→1 ou 1→0)
3. Reconvertir en hex → seulement le premier chiffre change

**Pour expliquer pourquoi deux repr. hex diffèrent beaucoup :**
- Vérifier si les deux nombres sont de part et d'autre d'une puissance de 2
- Si oui → l'exposant change → la virgule flotte → la mantisse change
  complètement → les repr. hex ont peu en commun

## ✏️ Application — Exercice 14

**Énoncé :**

(a) Expliquer pourquoi on n'utilise pas `00000000000` pour l'exposant
le plus petit et `11111111111` pour l'exposant maximal.

(b) Sachant que `format hex; pi` donne `400921fb54442d18`,
écrire le résultat de `format hex; -pi`.

(c) `format hex; pi+1` donne `401090fdaa22168c`.
Expliquer pourquoi $\pi$ et $\pi+1$ ont des repr. hex si différentes.

---

### Partie (a) : Exposants réservés

Les exposants `00000000000` et `11111111111` sont **réservés** pour des
valeurs spéciales que la représentation normale ne peut pas exprimer :

| Exposant | Mantisse | Valeur spéciale |
|---|---|---|
| `00000000000` | `000...0` | **Zéro** ($+0$ ou $-0$) |
| `00000000000` | autre | **Nombres dénormalisés** (très petits) |
| `11111111111` | `000...0` | **$\pm\infty$** (overflow) |
| `11111111111` | autre | **NaN** (résultat impossible : $0/0$, $\sqrt{-1}$...) |

Si on utilisait ces combinaisons pour des nombres normaux, on **perdrait**
la capacité de représenter 0, $\infty$ et NaN — ce qui est inacceptable.

---

### Partie (b) : Représentation de $-\pi$

```
 π = 400921fb54442d18
     ↑
     4 = 0100  ← premier bit = 0 → positif

-π = c00921fb54442d18
     ↑
     c = 1100  ← premier bit = 1 → négatif
```

**Seul le premier chiffre hex change** : `4` (=`0100`) → `c` (=`1100`).
Les 15 chiffres restants sont identiques.

---

### Partie (c) : Pourquoi $\pi$ et $\pi+1$ diffèrent tant

```
       2¹=2      π≈3.14    2²=4   π+1≈4.14    2³=8
        |           |        |        |          |
────────┼───────────┼────────┼────────┼──────────┼────
```

$$\pi \approx 3.14 = \mathbf{1.5707...} \times 2^{\mathbf{1}}$$

$$\pi + 1 \approx 4.14 = \mathbf{1.0354...} \times 2^{\mathbf{2}}$$

Les exposants sont différents (1 vs 2) → la virgule a "flotté" d'une
position → les 52 bits de mantisse encodent des plages numériques
différentes → presque aucun bit en commun → **seulement 3 chiffres
hex sur 16 sont identiques**.

> C'est exactement le phénomène qui donne son nom à la "virgule **flottante**".

---

### Réponse à écrire à l'exam :

> (a) Les exposants `00...0` et `11...1` sont réservés pour représenter
> les valeurs spéciales $0$, $\pm\infty$ et NaN, qui ne peuvent pas
> être exprimées par la représentation normale.
>
> (b) Seul le bit de signe change : `4` (= `0100`) → `c` (= `1100`).
> Résultat : **`c00921fb54442d18`**.
>
> (c) $\pi < 4 = 2^2$ mais $\pi+1 > 4 = 2^2$, donc les deux nombres
> n'ont pas le même exposant ($e=1$ vs $e=2$). La virgule flotte d'une
> position → les mantisses encodent des plages différentes → très peu
> de chiffres hex en commun.
